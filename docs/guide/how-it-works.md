---
title: 工作原理
---

这页讲 Crystal 如何在你的代码库里把事情做完：它如何组织上下文、调用模型、执行工具、收敛结束，以及你需要关注的安全边界与可调试性。这里不展开代码片段。

## Crystal 做什么

- 理解代码库：基于对话、文件内容与工具执行结果持续构建上下文
- 跨文件推进任务：不仅给建议，还能读/搜/改文件、执行命令，并把结果回写进上下文继续推进
- 以边界换确定性：默认保守、按需授权；每一步都可回溯

## 一次任务是怎么跑完的

一次会话由一系列回合组成。每一回合遵循固定节奏：

1. 组织上下文（必要时压缩，避免上下文过长导致失败）
2. 调用模型并流式输出
3. 从模型输出中解析工具调用
4. 执行工具，拿到结果（成功或失败都记录）
5. 把工具结果追加回对话历史
6. 若这一回合没有新的工具调用，任务收敛并结束；否则进入下一回合

## 分层结构（帮助扩展与调试）

- 交互层：负责 UI 呈现与输入处理
- 编排层：负责命令、会话生命周期、配置加载与日志落盘
- 核心循环：负责多轮推进与终止条件
- 工具层：负责文件/搜索/命令等本地动作与安全策略
- 模型层：负责与模型进行流式通信，并把响应标准化为上层可消费的事件

拆分的意义是：你可以替换模型或扩展工具，而不需要把整个系统改散；当出现异常时，也能快速定位问题发生在“哪一层”。

## 安全与权限（用户需要知道的部分）

Crystal 默认以更保守的方式运行：

- 写文件需要显式开启
- 执行命令需要显式开启

对使用者而言，记住一句话即可：默认安全，按需放权。

## 上下文压缩（为什么需要）

多轮任务里，工具输出与对话会快速变长。Crystal 会在必要时压缩上下文，以避免：

- 上下文窗口超限导致请求失败
- token 成本不受控

压缩的目标是保留关键事实：最近对话、关键工具结果、错误与路径信息，把历史折叠成可追溯的摘要。

## 可观测与可调试

每次会话会保存结构化日志（JSONL），用于复盘与定位问题：

- 模型的流式输出事件
- 工具调用与工具结果（含成功/失败）
- 回合边界与关键统计

建议把它当作“会话黑匣子”：当出现异常行为时，先看日志比猜 prompt 更有效。
